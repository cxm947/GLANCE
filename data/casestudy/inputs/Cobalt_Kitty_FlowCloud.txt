A spear-phishing link automatically redirected the victim to execute a PowerShell Cobalt Strike Beacon. A separate spear-phishing Word attachment created scheduled tasks (via a malicious macro) to pull further payloads from the C2 server. A VBScript loader then executed its commands, and its payload was first decoded from a base64/XOR blob. The decoded stage executed a PowerShell beacon in memory, completing the second delivery path into the same Cobalt Strike foothold. After the beacon executed, the attackers established persistence through Windows Registry Autorun keys. They hid the persisted payloads using NTFS Alternate Data Streams. They created and modified Windows Services to load the PowerShell beacon payloads. They also used a malicious Outlook backdoor macro for persistence by editing the LoadMacroProviderOnBoot value. A second set of scheduled tasks (a GoogleUpdate task) was used to run the malicious loaders on a timer. These loader-launching mechanisms — the Wsearch service and the scheduled task — ultimately performed DLL side-loading, placing a fake msfte.dll and goopdate.dll that the legitimate Windows Search and GoogleUpdate binaries loaded. The side-loaded loaders then used a regsvr32 COM-scriptlet to load additional Cobalt Strike payloads. For command and control, the fileless component used Cobalt Strike Malleable C2 profiles to impersonate Amazon and Google services through protocol impersonation. One C2 channel tunneled through DNS, relaying via Google and OpenDNS to reach the real C2 domain. Another C2 channel was an Outlook mail backdoor that relayed the attackers' commands through Gmail. After establishing C2, the attackers scanned the network for open ports and services. From that scanning they ran remote system discovery to enumerate the domain controllers. They also gathered network configuration with ipconfig and network connections with netstat. The reconnaissance led the attackers to dump credentials from LSASS using Mimikatz, obtaining NTLM hashes and Kerberos tickets. With those credentials they performed pass-the-hash attacks. Separately, they also carried out pass-the-ticket attacks. Using the stolen authentication material they spread via SMB/Windows Admin Shares. They also abused Windows Management Instrumentation (WMI) with the stolen credentials to deploy tools on remote machines.
